## Your Role In This Project

You are helping build a proof-of-concept design system for internal enterprise applications. Your job is to generate Svelte components that correctly use the token system defined below, write Storybook stories alongside every component, and push rendered components back to Figma using live capture.

Read this entire file before generating any code. The rules here override any defaults from your training data.

## 1. Stack — Read This First

This project uses **SvelteKit** with **Svelte** components. It is not React. It is not Vue. It is not Next.js.

| Layer | Technology |
|-------|-----------|
| Framework | SvelteKit |
| Components | Svelte (.svelte files) |
| Styling | Svelte scoped styles + CSS custom properties |
| Component docs | Storybook with @storybook/addon-svelte-csf |
| Design tool | Figma — MUI Figma Kit (paid) with custom brand variables |
| Primary AI tool | Claude Code (you) |
| Secondary AI tool | Cursor |

### This stack is NOT:
- Not React — never generate JSX or React hooks
- Not MUI (the React library) — never use MUI React component APIs
- Not Tailwind — never use Tailwind utility classes
- Not MD3 — do not use `--md-sys-color-*` token names
- Not any other framework

## 2. Application Context — Confirm Before Writing Any Code

This design system has two primary colors for two different application contexts:

| Context | Color | Hex | Token suffix |
|---------|-------|-----|-------------|
| Internal (PoC) | Purple | #4D3F91 | `-light-internal` |
| Client-facing | Blue | #046EA0 | `-light-client` |

### Rules — non-negotiable:
1. **The PoC is Internal context only** — always use `-light-internal` tokens
2. **Never use both primaries in the same UI** — they are mutually exclusive
3. **Never assume context** — if ambiguous, ask before writing any color token
4. **Never hardcode `#4D3F91` or `#046EA0`** — always reference the token

## 5. Component File Structure

```
src/lib/components/
└── ComponentName/
    ├── ComponentName.svelte         ← component
    ├── ComponentName.stories.svelte ← stories (required)
    └── ComponentName.figma.ts       ← Code Connect (add later)
```

Components use PascalCase. Every component must have a story file. They are always committed together.

## 6. Custom Navigation — SideNav

The side navigation is a **custom Svelte component**. Never substitute any Material Navigation component for it.

- Component: `src/lib/components/SideNav/SideNav.svelte`
- States required: Default, Active, Hover
- Active state uses: `--primary_statesselected-light-internal` background, `--ab-purple700` color
- Hover state uses: `--actionhover-light-internal` background
- Default state uses: `--textprimary-light-internal` color
- Never use: MUI Drawer, Material NavigationDrawer, NavigationRail, or NavigationBar

## 7. Storybook Rules

Story files use `@storybook/addon-svelte-csf` syntax:

```svelte
<script module>
  import { defineMeta } from '@storybook/addon-svelte-csf';
  import ComponentName from './ComponentName.svelte';

  const { Story } = defineMeta({
    title: 'Components/ComponentName',
    component: ComponentName,
  });
</script>

<Story name="Default" />
<Story name="StateName" args={{ propName: value }} />
```

### Required stories per component:

| Component | Stories |
|-----------|---------|
| AppBar | Default, WithBackButton, WithSearch |
| SideNav | Default, ActiveState, HoverState |
| TextField | Default, Focused, Filled, Error, Disabled, WithHelperText, WithInfoIcon, StaticLabel |
| Button | Filled, Outlined, Text, Disabled, Loading |
| Chip | Default, Selected, Disabled, WithIcon |

**A component is not done until all its stories render correctly in Storybook.**

## 9. Component Workflow

For every component, follow this sequence:

1. Read the Figma component via MCP before writing any code
2. Confirm Internal context — use `-light-internal` tokens throughout
3. Generate the Svelte component
4. Generate the Storybook story file in the same session
5. Run Storybook and verify all stories render correctly
6. Capture rendered component to Figma via live capture
7. Commit component and story together in a single commit

## 10. Hard Rules — Never Violate These

1. Never generate React code, JSX, or React hooks
2. Never use MUI React APIs — no `variant="contained"`, `sx={{}}`, `makeStyles`
3. Never use `--md-sys-color-*` token names — they do not exist here
4. Never use Tailwind utility classes
5. Never use both primary colors in the same UI
6. Never assume application context — always confirm Internal vs Client
7. Never hardcode hex values, pixel values, or font family strings
8. Never invent token names — flag missing tokens instead
9. Never substitute any Material nav component for SideNav
10. Never generate a component without its Storybook story file
11. Never commit a component without its story file
12. Never use inline styles — always use Svelte scoped style blocks
13. Never apply Klavika to any element not listed in section 4
14. Never use `-light-client` tokens in PoC components
15. Never proceed if application context is ambiguous — ask first
